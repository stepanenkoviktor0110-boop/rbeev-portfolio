# PROJECT_DOCS

Техническая документация по текущему состоянию проекта `LevelUPMe`.

## 0. Архитектурный ориентир и текущее состояние

- Цель проекта: любой контентный блок на публичном сайте должен управляться через админ-панель.
- Текущее исключение: блок `HowItWorks` на витрине использует фиксированные тексты карточек и фиксированную строку стоимости.
- Поле `Settings.workflowItems` и UI редактирования в админке пока сохраняются в БД, но публичной страницей не используются.
- Исправлены проблемы кодировки UI-строк (mojibake) в:
  - `app/(auth)/login/page.tsx`
  - `components/CookieConsentBanner.tsx`
  - `app/gallery/[id]/page.tsx`

## 1. Стек

- `Next.js 14` (App Router)
- `React 18`
- `TypeScript`
- `Tailwind CSS`
- `Prisma` + `PostgreSQL`
- `Yandex Disk API` (хранение и публикация изображений)

Привязка storage:

- Проект работает с одним конкретным Яндекс.Диском, который определяется `YANDEX_DISK_TOKEN`.
- Все серверные операции с фото (загрузка/публикация/удаление) выполняются в этом аккаунте.
- Базовый путь хранения задается `YANDEX_DISK_BASE_DIR`; для токенов с доступом к папке приложения используйте формат `app:/site-gallery`.

## 3. Публичная часть

### 3.1 Hero

- Заголовок + подзаголовок + кнопка CTA.
- Третья строка микрокопи под CTA удалена.
- CTA рамочная (`border 1.5px`), без заливки.
- Фон: слайдшоу из фото `showInSlideshow`.

### 3.2 Лента доступности дат (`AvailabilityRibbon`)

Расположение: между `Hero` и `Gallery`.

Текущее поведение:

- строит 10 дат от завтрашнего дня
- локализует день недели/месяц на русском
- показывает заголовок «Мой календарь съёмок» + увеличенный месяц/год
- свободная дата кликабельна
- занятая дата не кликабельна, с приглушенным стилем
- hover для свободной даты: золотой акцент, подсказка

Интеграция с формой:

- при клике на свободную дату:
  - `scrollIntoView` к `#contacts`
  - dispatch события `shooting-date:selected` с `isoDate`
- `ContactForm` слушает событие и:
  - подставляет дату в поле
  - подсвечивает поле на 1 секунду

Источник занятых дат:

- серверный запрос в таблицу `BusyDate` (поле `isoDate`)
- формат хранения: `YYYY-MM-DD`

### 3.3 Галерея

- Фильтры по категориям.
- Сетка карточек (grid).
- Hover на фото: только легкое затемнение.
- Для режима `Все` действует лимит: показываются только первые 12 фото.
- Если фото больше 12, в конце галереи показывается длинная узкая кнопка-индикатор: `Еще N фото`.
- Кнопка «Смотреть все работы» не используется в режиме `Все` (чтобы не раскрывать весь список сразу).
- Порядок карточек в режиме `Все`: порядок категорий (`Category.sortOrder`) -> порядок фото в категории (`Photo.sortOrder`).

### 3.4 Лайтбокс (`Lightbox`)

- Минималистичный оверлей без заголовка/счетчика/крестика.
- Закрытие кликом вне фото.
- Нециклическая навигация:
  - стрелки показываются только если есть куда идти
  - на первом фото нет «влево», на последнем нет «вправо»
- Поддержка:
  - клавиши `ArrowLeft/ArrowRight`
  - свайп на мобильных
  - клик по самому фото -> следующее; на последнем фото -> закрыть

### 3.5 Обо мне

- Без стрелок и dots.
- Фото переключаются автокроссфейдом (медленно).
- Ссылка «Перейти к форме» в виде текстовой ссылки без рамки.

### 3.6 Как всё устроено

- 3 колонки (не 4).
- секция имеет якорь `#how-it-works`.
- Тексты карточек фиксированы:
  - 01: «Приеду туда, где вам будет комфортно — парк, двор, любимое место.»
  - 02: «Не тороплю. Снимаем столько, сколько нужно.»
  - 03: «Фотографии пришлю в течение недели.»
- Ниже отдельной строкой:
  - «Стоимость — от 15 000 ₽. Детали обсудим, когда напишете.»
  - `14px`, `rgba(255,255,255,0.45)`, выравнивание влево
- `Settings.workflowItems` в текущей версии не влияет на публичный вывод этого блока.

### 3.7 Отзывы

- После блока контактов/формы добавлен блок «Отзывы».
- Публичный блок получает отзывы из БД (`Review`) и показывает только `isPublished = true`.
- Порядок на витрине задается через `sortOrder`.
- Если опубликованных отзывов нет, блок полностью скрывается на публичной странице.

### 3.8 Форма контактов

- Inline-валидация с визуальной отметкой заполнения.
- Плейсхолдер сообщения:
  - «Расскажите — кто, по какому поводу, есть ли идеи по локации»
- Чекбокс: «Согласен на обработку данных».
- После отправки показывается экран благодарности.

## 4. Админ-панель

### 4.1 Разделы

- `Галерея`
- `Категории`
- `Заявки`
- `Отзывы`
- `Настройки`

### 4.1.1 Галерея: режимы загрузки фото (актуально)

- В разделе `Админка -> Галерея` доступны 2 режима импорта через Яндекс.Диск:
  - загрузка файлов `1..10` за одну операцию (как раньше, но storage теперь Яндекс.Диск)
  - импорт по публичной ссылке на папку Яндекс.Диска
- Для импорта папки:
  - создается категория с именем папки (или используется существующая с тем же именем)
  - все найденные изображения импортируются и получают `showInGallery = true`, поэтому сразу отображаются на витрине.
  - ссылка должна быть публичной; внутренние ссылки формата `/client/disk/...` отклоняются.
- Прогресс в админке:
  - для загрузки файлов (`1..10`) прогресс считается по реально загруженным байтам (`XMLHttpRequest.upload.onprogress`);
  - для импорта папки прогресс идет с сервера по реально выполненным шагам (перенос файлов из публичной папки и последующая запись в БД).
- В форме загрузки файлов вся карточка является зоной `drag&drop` и клика для выбора файлов.

### 4.1.2 Галерея: визуал прогресса импорта папки (обновлено)

- В форме `Админка -> Галерея -> Импорт папки по ссылке` прогресс показывается в отдельном читаемом блоке:
  - верхний общий progress bar + числовой процент;
  - текущая текстовая подпись шага (например, подготовка/перенос/сохранение);
  - две отдельные строки этапов:
    - `Перенос файлов`
    - `Сохранение в БД`
- Для каждого этапа отображается:
  - счетчик выполнения (`completed / total`);
  - статус этапа: `Ожидание`, `В процессе`, `Готово`.
- Логика статусов:
  - `Перенос файлов`: `Ожидание` на фазе `start`, `В процессе` на фазе `transfer`, `Готово` после перехода к `db` и при `done`;
  - `Сохранение в БД`: `Ожидание` до фазы `db`, `В процессе` на фазе `db`, `Готово` при `done`.
- Общий процент импорта для папки считается как комбинированный прогресс двух стадий (перенос + БД), где суммарный объем работ равен `total * 2`.

### 4.2 Занятые даты (актуально)

Управление занятыми датами выполняется в `Админка -> Заявки`.

Что есть в UI:

- календарь ближайших 10 дней (вверху страницы)
- клик по свободному дню предлагает создать заявку на выбранную дату
- без заявки занять дату нельзя
- клик по занятому дню снимает занятость
- в карточке заявки кнопка «Подтвердить дату в календаре»
- в форме редактирования заявки эта же кнопка доступна

Сохранение:

- через `POST /api/busy-dates` (создание связи `дата -> заявка`)
- через `DELETE /api/busy-dates/[isoDate]` (снятие занятости)

Важно по дате:

- применена локальная нормализация `YYYY-MM-DD` без `toISOString()`
- это убирает сдвиг на +1 день из-за timezone

### 4.3 Настройки

- в `Настройках` нет управления занятыми датами
- здесь остаются тексты/контакты/hero/workflow/о себе/смена пароля
- важно: `workflowTitle` применяется на витрине, `workflowItems` пока только сохраняется
- пароль админки в БД хранится только в виде `bcrypt`-хэша (`Settings.adminPasswordHash`)

#### 4.3.1 Сброс пароля админки

Если пароль утерян, его нельзя прочитать из БД в открытом виде; можно только сбросить.

Порядок сброса:

1. Задать новый `ADMIN_PASSWORD` в `.env`.
2. Для `Settings(id=1)` очистить `adminPasswordHash` (или удалить запись `Settings`).
3. При следующей проверке логина приложение создаст новый хэш из `ADMIN_PASSWORD`.

### 4.4 Категории (актуально)

- Полный цикл в админке: создание, редактирование, удаление.
- При удалении категории с привязанными фото показывается подтверждение.
- Добавлено ручное перемещение категорий стрелками вверх/вниз.
- Порядок категорий применяется и в админке, и в публичной галерее.

### 4.5 Производительность (выполненные оптимизации)

- Категории: после create/edit/move/delete список обновляется локально, без полного рефетча.
- Заявки: календарь занятых дат обновляется локально после assign/unassign/delete.
- Галерея: при удалении фото избегается лишний полный рефетч страницы, если это не требуется.
- Витрина:
  - `next/image` используется для Hero и карточек галереи;
  - карточки галереи загружаются лениво;
  - в блоке «Обо мне» рендерится только текущий кадр слайдера (без одновременного рендера всех кадров).

### 4.6 Отзывы (актуально)

- В админке добавлен раздел `Отзывы`.
- Поддерживается:
  - создание отзыва,
  - редактирование автора/текста,
  - удаление,
  - публикация/скрытие (`isPublished`),
  - изменение порядка (вверх/вниз через `sortOrder`).

## 5. Prisma

`prisma/schema.prisma`:

- `Photo`:
  - `imageUrl: String` (публичный URL изображения)
  - `storageKey: String?` (внутренний путь ресурса в Яндекс.Диске для удаления/служебных операций)
- `BusyDate`:
  - `isoDate: String @unique`
  - `requestId -> ContactRequest.id`
- `Category.sortOrder: Int @default(0)`
- `Review`:
  - `authorName: String`
  - `text: String`
  - `sortOrder: Int @default(0)`
  - `isPublished: Boolean @default(true)`
- `ContactRequest.shootingDate: String`
- `Settings.busyDates` остается в схеме только как legacy-поле (не используется в новом календаре)

Миграции для локального окружения:

```bash
npx prisma generate
npx prisma db push
```

Для существующей БД со старыми данными рекомендуется применять SQL-миграции:

```bash
npx prisma migrate deploy
```

## 6. API (ключевое)

- `GET/PUT /api/settings`
  - чтение/сохранение настроек (без управления занятыми датами)
- `GET/POST /api/requests`
  - список и создание заявок
- `PUT/DELETE /api/requests/[id]`
  - редактирование/удаление заявки
- `GET/POST /api/busy-dates`
  - список занятых дат и создание записи занятости с привязкой к заявке
- `DELETE /api/busy-dates/[isoDate]`
  - удаление занятости по дате
- `GET/POST/PUT/DELETE /api/categories`
  - полный CRUD + reorder + удаление с предупреждением при связанных фото
  - при reorder (`up/down`) `sortOrder` пересобирается последовательно (`1..N`) для стабильного порядка
- `GET/POST /api/reviews`
  - список отзывов (админ), создание
- `PUT/DELETE /api/reviews/[id]`
  - обновление и удаление отзыва
- `GET/POST /api/photos`
  - список фото (админ)
  - загрузка файла в Яндекс.Диск и создание записи `Photo`
- `POST /api/photos/import-folder`
  - импорт изображений из публичной папки Яндекс.Диска
  - автосоздание/поиск категории по имени папки
  - ссылки вида `/client/disk/...` не принимаются (требуется публичная ссылка)
  - ответ идет потоково (`application/x-ndjson`) с событиями прогресса:
    - `progress` (`phase: start|transfer|db`)
    - `done`
    - `error`
- `PUT/DELETE /api/photos/[id]`
  - редактирование метаданных фото
  - удаление фото из БД + best-effort удаление файла из Яндекс.Диска по `storageKey`

## 7. Проверка после изменений

Рекомендуемый минимум:

1. Главная открывается без runtime-ошибок.
2. Клик по свободной дате скроллит к форме и подставляет дату.
3. В админке (`Заявки`) подтверждение даты из карточки помечает правильный день.
4. Лайтбокс:
   - листает вперед/назад клавишами
   - клик по фото листает вперед
   - на последнем фото клик закрывает
   - нет циклического перехода с последнего на первое.
5. Админка -> Галерея:
   - загрузка 1..10 файлов успешно создает фото в выбранной категории.
6. Импорт папки Яндекс.Диска:
   - создается/используется категория с именем папки
   - импортированные фото сразу видны на витрине (включен `showInGallery`).
   - в прогрессе импорта видны отдельные этапы `Перенос файлов` и `Сохранение в БД` со счетчиками и корректными статусами (`Ожидание` / `В процессе` / `Готово`).
